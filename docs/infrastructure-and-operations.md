# Инфраструктура и эксплуатация (P0)

Документ закрывает репозиторийную часть блока **P0 — инфраструктура и эксплуатация** из `docs/launch-backlog.md` (раздел 2). Развёртывание managed-сервисов и подключение внешних алертов выполняются командой на стороне облака; здесь — чеклисты, порядок действий и ссылки на runbook.

## 1. Staging, близкий к prod

Цель: те же классы зависимостей и те же переменные окружения (по смыслу), что в production, чтобы ловить миграции Prisma, webhooks платежей и загрузку файлов до релиза.

| Компонент | Требование | Примечание |
|-----------|------------|------------|
| Postgres | Отдельный инстанс, не общий с prod | Те же major-версии и параметры пула, что в prod |
| Redis | Отдельный инстанс | Для BullMQ очередей `notifications` / `appointments` |
| Object storage | S3-совместимое API (MinIO в dev, managed bucket в staging) | URL бакета и ключи только в секретах CI/рантайма |
| Секреты | Вне git: менеджер секретов или sealed secrets | Зеркалировать имена ключей из `.env.example`, не значения |
| API | Один deploy API + воркеры очередей | Те же образы/теги, что кандидат в prod |

После поднятия среды: применить миграции (`prisma migrate deploy`), прогнать smoke из раздела 3.

## 2. Production

| Компонент | Требование |
|-----------|------------|
| Postgres | Managed: автоматические бэкапы; по возможности **PITR** (point-in-time recovery) |
| Redis | Managed или кластер с persistence по политике SLA |
| Object storage | Версионирование/политики удаления по согласованию; prod-бакет отдельно от staging |
| Секреты | Только во внешнем хранилище; ротация без коммита в репозиторий |
| Воркеры | Процесс(ы) BullMQ для тех же очередей, что в dev — иначе `health/queues` и напоминания не работают |

## 3. Smoke после деплоя

Выполнить сразу после выката (staging или prod), до объявления релиза «зелёным».

1. **Liveness:** `GET /api/v1/health/live` → `200`, тело `status: ok`.
2. **Зависимости:** `GET /api/v1/health` → `200`, `postgres` и `redis` в `up`, очереди `notificationsQueue` / `appointmentsQueue` в `up`.
3. **Очереди и пороги:** `GET /api/v1/health/queues` → `200`, поле `status` в `ok` | `warn` | `critical`; при первом деплое допустим `ok` при пустых очередях. Пороги задаются env `ALERT_*` (см. `.env.example`).
4. **Критичный бизнес-smoke** (по scope релиза): минимум один сценарий из релизного чеклиста `docs/release-checklist.md` (например создание записи / оплата на staging).

Заголовок `X-Request-Id` в ответах помогает сопоставить запрос с логами.

## 4. План отката

### Приложение

1. Переключить трафик (или deployment) на **предыдущий известный стабильный** образ/тег API и воркеров.
2. Перезапустить воркеры очередей после отката.
3. Повторить smoke из раздела 3.

### Миграции базы

- По правилам репозитория схема меняется только через **Prisma migrations**; порядок rollout: additive-изменения → код → при необходимости backfill/cleanup.
- **Обратимость:** для каждой миграции в PR фиксируйте, безопасен ли откат только приложения без отката БД. Destructive-шаги (удаление колонок, сужение enum) — только после окна наблюдения и бэкапа.
- Если после миграции нужен откат схемы: либо заранее подготовленная down-мigration (редко в Prisma по умолчанию), либо **восстановление БД из бэкапа/PITR** по runbook DBA — заранее согласовать RTO/RPO.

Краткий операционный откат при инцидентах очередей: `docs/reminders-lifecycle-runbook.md`, раздел 5.

## 5. Алерты по `GET /api/v1/health/queues`

Эндпоинт **публичный** (без JWT): его можно дергать внутренним мониторингом из VPC или через защищённый прокси с IP allowlist — **не** выставляйте без защиты в публичный интернет, если в ответе есть чувствительные агрегаты (при необходимости ограничьте доступ на балансере).

**Что алертить:**

- HTTP не `200`.
- JSON: `status === "warn"` → уведомление дежурным; `status === "critical"` → пейджинг по политике команды.
- Дополнительно: рост `failed` в `GET /api/v1/health` по счётчикам очередей.

**Пороги** приходят в теле ответа (`alerts.thresholds`) и задаются env:

- `ALERT_REMINDER_DELAYED_WARN` / `ALERT_REMINDER_DELAYED_CRITICAL`
- `ALERT_LIFECYCLE_DELAYED_WARN` / `ALERT_LIFECYCLE_DELAYED_CRITICAL`

Пример проверки вручную:

```bash
curl -sS "https://<api-host>/api/v1/health/queues" | jq '.status, .alerts.breaches'
```

Инцидентный runbook: `docs/reminders-lifecycle-runbook.md`.
