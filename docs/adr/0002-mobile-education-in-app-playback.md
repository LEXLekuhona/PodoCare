# ADR 0002: Раздел «Обучение» — просмотр видео внутри приложения

**Статус:** принято (направление развития)  
**Дата:** 2026-05-07

## Контекст

- В мобильном клиенте есть раздел **«Обучение»**: лента материалов, paywall, прогресс, CTA.
- По продукту основной формат материалов — **видео**.
- Нужно зафиксировать, остаётся ли пользователь в приложении при просмотре, и согласовать последующий план работ.

## Решение

- **Основной сценарий:** пользователь **остаётся в приложении**: отдельный экран материала с **нативным видеоплеером** по данным, приходящим с API (прямой URL на медиа, при необходимости позже — HLS).
- **Внешние ссылки** (`Linking.openURL`, CTA с `EXTERNAL_URL`) остаются **допустимыми вторичным путём**: оплата, лендинг вебинара, партнёрский сайт и т.п. — но **не** как основной способ показа обычного урока/видео из каталога обучения.

## Контракт данных (целевой)

Для `ContentItem` с `format: VIDEO` в **`body`** (JSON) как минимум:

- `videoUrl` — HTTPS URL, пригодный для нативного плеера (на старте обычно MP4).
- Опционально `hlsUrl` — когда появится адаптивная выдача.

На сущности (уже есть в API):

- `coverImageUrl` — обложка в списке и на экране материала.
- `durationSeconds` — для UI и валидации прогресса.

Клиент не должен полагаться на «магические» поля без явного контракта; при расширении схемы — версионирование или документирование в этом ADR.

## План работ по фазам

### Фаза 1 — мобильный клиент: экран материала и плеер

- Маршрут (expo-router), например экран детали по `itemId` из ленты.
- Плеер: **`expo-av`** или **`expo-video`** в зависимости от версии Expo в репозитории; полноэкран по необходимости, понятные состояния загрузки и ошибки (сеть, 403, неподдерживаемый формат).
- Переход с **«Обучение»**: тап по карточке материала открывает экран просмотра (не только кнопка CTA).

### Фаза 2 — прогресс и возобновление

- Связать **`saveContentItemProgress`** (и при необходимости `lastPositionSeconds`) с событиями плеера (throttle, пауза/закрытие экрана).
- Отображение процента в ленте согласовать с фактическим просмотром.

### Фаза 3 — админка

- Ввод материала **VIDEO** через форму (URL видео, опционально HLS, обложка, длительность), а не через сырой JSON `body`, чтобы снизить ошибки редакторов.

### Фаза 4 — инфраструктура (по мере нагрузки)

- Надёжный хостинг/CDN для видео; при необходимости загрузка через presigned URL; позже — транскодинг и единый HLS-поток при росте требований к качеству в мобильных сетях.

## Последствия

- **Плюсы:** единый UX, контроль доступа (paywall), проще аналитика просмотра, пользователь не «теряется» во внешнем браузере.
- **Минусы:** зависимость от качества URL/кодеков/HTTPS; нужно явно обрабатывать ошибки и офлайн; возможны ограничения платформ по фоновому воспроизведению и PiP (уточнять при реализации).

## Альтернативы (отклонены для базового видео)

- **Основной просмотр только через внешний сайт** (браузер) — хуже удержание и контроль продукта.
- **Только WebView с чужим плеером** как основной путь — лишняя непредсказуемость (автоплей, полноэкран, cookies) без явной необходимости.
